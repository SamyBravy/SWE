    \section{Description}
    In this project, we developed a platform for using and sharing artificial intelligence models among users, who are divided into three roles:
    \begin{itemize}
        \item \textbf{Model Provider:} a user who publishes one or more models on the platform for profit, deciding the cost per token they want to receive from the usage of their models.
        \item \textbf{Developer:} a user who can use any model on the platform. They have a balance in their wallet, to which the used tokens are gradually charged. The platform also allows them to submit complaints, in case there is any issue.
        \item \textbf{Supervisor:} a user who moderates the platform, verifying the models uploaded by the Model Provider before they are actually published, and resolving complaints.
    \end{itemize}

    The platform adds a fee to the model's price to cover its hosting costs. 
    Consequently, we added a layer of complexity to our project by designing a system to manage model hosting across multiple GPUs,
    which have a limited memory and a temperature that needs to be regulated. 
    \section{Project Setup and Technologies}
    \label{sec:Architecture}
    This project was developed using the \texttt{Java 17} programming language, managed through Maven for dependency and build management. 
	We integrated two external libraries: \texttt{jBCrypt} for password hashing, and \texttt{JUnit 5} for unit testing. 
	We used GitHub for version control and collaboration, StarUML for the use case diagram and package diagram, PlantUML for the class diagram, and Mermaid for the page navigation diagram.
    \\ We used AI for the controllers, particularly to generate outputs and menus, and for the creation of mockups.   
	\\ We did not implement a real database, but we simulated persistence using some basic hashmaps.
    \\ We divided the software into four main parts:
    
    \begin{itemize}
        \item \textbf{Domain Model:} contains the classes that define the entities involved in the system and their relationships.
		\item \textbf{Controllers:} contains the classes that handle the user input and delegate actions to the business logic.
        \item \textbf{Business Logic:} contains the classes that define the rules and the logic of the system, regardless of how the data is stored.
        \item \textbf{ORM:} contains the classes that communicate with the database, to make the changes in the domain model persistent.
    \end{itemize}
    
    Due to the exam's scope and the software's complexity, the platform abstracts many details, such as the GPUs and the models, and does not implement a central server to enable 
    communication between the users and the different parts of the system. Furthermore, for the interface, we used a CLI instead of a GUI, for which we only built a few mockups.

    \subsection{Design Specifications}
    To ensure high cohesion, low coupling, and overall software maintainability, we implemented several design patterns.
    
    \subsubsection{Observer Pattern}
    The Observer pattern establishes a dependency between objects, ensuring that when one object changes state, all its dependents are notified and updated accordingly. In this implementation, it allows the \texttt{LoadBalancerService} to monitor system-wide changes dynamically. Specifically, the service acts as a Concrete Observer to two distinct subjects: the \texttt{Timer} and the \texttt{GPU} classes.
    \smallskip
    
    The \texttt{Timer} serves as the system’s central clock. Implemented as a Singleton (see later ~\vref{subsub:Singleton}) to ensure a single, synchronized source of time, it uses an internal thread to execute at fixed intervals. Every cycle, the Timer broadcasts a generic "TICK" event. Upon receiving this signal, the \texttt{LoadBalancerService} triggers its \texttt{balanceLoad()} routine to periodically reshuffle the workload and ensure optimal distribution.
    
    Simultaneously, the \texttt{GPU} acts as a second subject. It monitors hardware health and notifies its subscribers if its temperature reaches the critical threshold. This allows the \texttt{LoadBalancerService} to react not just to time, but to urgent hardware demands, triggering an immediate load redistribution to mitigate overheating.
    Figure ~\ref{fig:Obspattern} visually explains what has been said.

    \subsubsection{Strategy Pattern}
    The Strategy pattern is employed to define a family of algorithms, encapsulate each one, and make them interchangeable (see Figure~\ref{fig:StrategyGeneral}). This pattern is primarily used within the load balancing mechanism.
    \smallskip
    
    The system utilizes a \texttt{LoadBalancingStrategy} interface (see Figure~\ref{fig:Strategy}) to expose load-balancing capabilities without tying the system to a single implementation.

	Two distinct algorithms are implemented: the \texttt{EvenBalancingStrategy}, which distributes workloads equally across available hardware, and the \texttt{EcoBalancingStrategy}, which focuses on power-saving.
	
	This allows the choice of the balancing algorithm to be determined dynamically without modifying the underlying service logic. Furthermore, as the system evolves, new balancing strategies can be added arbitrarily by simply implementing the existing interface.

    \subsubsection{Singleton Pattern}
    \label{subsub:Singleton}
    The Singleton pattern restricts the instantiation of a class to a single, globally accessible object. This is critical for managing shared, system-wide resources to prevent conflicting states.
    \smallskip
    
    The \texttt{GpuCluster} is implemented as a singleton to serve as the centralized manager for all GPUs. To this end, as seen in listing ~\vref{lst:Singleton}, the constructor is declared private so that it can only be invoked by methods within the class itself.
    
    The class therefore provides a static method as an access point, which can be called before the object has been instantiated. Consequently, the design pattern involves maintaining a static attribute to be used both as a pointer to the object and as a flag indicating whether instantiation has already occurred. This attribute must be static, as a static method can only access attributes that are also static. Furthermore, in this case, this static method is synchronized to ensure thread safe access.
    
    \begin{minipage}{\linewidth}
    	\begin{lstlisting}[caption={GpuCluster singleton design pattern implementation}, label = {lst:Singleton}]
    		public class GpuCluster {
    			
    			private static GpuCluster instance;
    			...
    			
    			private GpuCluster() {
    			}
    			
    			public static synchronized GpuCluster getInstance() {
    				if (instance == null) {
    					instance = new GpuCluster();
    				}
    				return instance;
    			}
    			...
    		}
    	\end{lstlisting}
    \end{minipage}

    \smallskip
    The \texttt{Timer} also follows the singleton pattern to ensure that only one global clock dictates the simulation's tempo, preventing erratic or overlapping tick events.

	\subsection{User}
	The \texttt{User} class is an abstract class that represents a user of the platform. It's worth noticing 
	the implementation of the \texttt{login} method and \texttt{setPassword} method that use the \texttt{BCrypt} library to implement the password hashing.
	\subsubsection{\texttt{setPassword} and \texttt{login}}
	\begin{lstlisting}[label={lst:userauth}]
	public final void setPassword(String password) {
		this.password = BCrypt.hashpw(password, BCrypt.gensalt());
	}
	
	public boolean login(String email, String pwd) {
		return this.email.equals(email) && BCrypt.checkpw(pwd, this.password);
	}
	\end{lstlisting}
	The method \texttt{BCrypt.checkpw} verifies that the hash of the password provided by the user is equal to the hash stored in the database,
	whereas \texttt{BCrypt.hashpw} hashes the password.
	
	
	\subsection{Developer, ModelProvider, Supervisor}
	These three classes extend the abstract \texttt{User} class.
	They implement the \texttt{getRole()} method to return their specific role as a string. 
	The \texttt{Developer} class is the only one that accepts a \texttt{Wallet} object in its constructor in order to handle payments for using AI models.
	
	\subsection{AiModel}
	This class represents an AI model published by a Model Provider. 
	The \texttt{costPerTokenProvider} is the cost per token chosen by the Model Provider, whereas the \texttt{costPerTokenPlatform} is the cost per token for the hosting decided by the Supervisor; the Developer will pay the sum of these two values.
	We used the static factory method \texttt{submitForReview} for a robust validation of the passed arguments and to make the constructor private, since only the Model Provider can create a new model when he submits it for review.
	\subsubsection{\texttt{submitForReview}}
	\begin{lstlisting}[label={lst:aimodelsubmit}]
	public static AiModel submitForReview(int id, String name, String description, double costPerTokenProvider,
	                                      String safetensorsFile, String jsonFile, ModelProvider provider) {
		if (id <= 0) {
			throw new IllegalArgumentException("Model ID must be positive. Received: " + id);
		}
		if (name == null || name.trim().isEmpty()) {
			throw new IllegalArgumentException("Model name cannot be null or empty");
		}
		// ... other validations ...
		if (safetensorsFile == null || !safetensorsFile.toLowerCase().endsWith(".safetensors")) {
			throw new IllegalArgumentException("Valid .safetensors file path is required");
		}
		if (jsonFile == null || !jsonFile.toLowerCase().endsWith(".json")) {
			throw new IllegalArgumentException("Valid .json file path is required");
		}
		Objects.requireNonNull(provider, "Model provider cannot be null");
	
		return new AiModel(id, name, description, costPerTokenProvider, safetensorsFile, jsonFile, provider);
	}
	\end{lstlisting}
	\subsection{Complaint}
	This class represents a complaint submitted by a Developer against a model. 
	The \texttt{promptLogs} list is used so that the Supervisor can decide, reading the logs, whether to accept or reject the complaint.
	Like we did in \texttt{AiModel}, we used a static factory method for the same reasons, since a Developer can create a new complaint only when he submits it through the \texttt{submit} method.
	
	\subsubsection{\texttt{submit}}
	\begin{lstlisting}[label={lst:complaintsubmit}]
	public static Complaint submit(int id, Developer developer, AiModel model, String description, List<String> promptLogs) {
		if (id <= 0) {
			throw new IllegalArgumentException("Complaint ID must be positive");
		}
		Objects.requireNonNull(developer, "The developer filing the complaint cannot be null");
		Objects.requireNonNull(model, "The model being reported cannot be null");
	
		if (description == null || description.trim().isEmpty()) {
			throw new IllegalArgumentException("Complaint description cannot be empty");
		}
	
		Objects.requireNonNull(promptLogs, "Prompt logs cannot be null");
		if (promptLogs.isEmpty()) {
			throw new IllegalArgumentException("A complaint must include at least one prompt log as evidence");
		}
	
		return new Complaint(id, developer, model, description, promptLogs);
	}
	\end{lstlisting}
	
	\subsection{Session}
	This class acts as a bridge when a Developer interacts with a model. 
	It maintains a list of GPUs where the associated model is running, allowing load balancing across multiple GPUs. 
	Furthermore, it keeps a history of sent and received prompts so that, in case of a complaint, the Supervisor can access them as evidence. 
	Finally, the \texttt{totalTokensUsed} and \texttt{startTimestamp} attributes are used to record statistics.
	
	\subsection{Wallet and Transaction}
	The \texttt{Wallet} class handles the Developer's balance that is used to pay for the used tokens. Every time the balance is modified, a new \texttt{Transaction} is created and added to the \texttt{transactionHistory} list.
	For the \texttt{Transaction} class we used the static factory method \texttt{create} to ensure a transaction is always created completely and correctly. The \texttt{Transaction} class is immutable, as all its attributes are \texttt{final} and it exposes only getter methods.
	
	\subsubsection{\texttt{create}}
	\begin{lstlisting}[label={lst:transactioncreate}]
	public static Transaction create(int id, double amount, LocalDateTime timestamp, String reason) {
		if (id <= 0) {
			throw new IllegalArgumentException("Transaction ID must be positive...");
		}
		if (Double.isNaN(amount) || Double.isInfinite(amount)) {
			throw new IllegalArgumentException("Invalid amount: " + amount);
		}
		Objects.requireNonNull(timestamp, "Timestamp cannot be null");
		if (reason == null || reason.trim().isEmpty()) {
			throw new IllegalArgumentException("Transaction reason cannot be null or empty");
		}
		return new Transaction(id, amount, timestamp, reason);
	}
	\end{lstlisting}
	
	The \texttt{Wallet} class uses the \texttt{addFunds} and \texttt{charge} methods to modify the balance. 
	\subsubsection{\texttt{addFunds}}
	\begin{lstlisting}[label={lst:walletaddfunds}]
	public void addFunds(double amount, String... optionalReason) {
		if (amount <= 0) {
			throw new IllegalArgumentException("Amount must be positive");
		}
		String reason = (optionalReason != null && optionalReason.length > 0)
				? optionalReason[0]
				: "TOP-UP";
	
		this.balance += amount;
		transactionHistory.add(Transaction.create(
				nextTransactionId++, amount, LocalDateTime.now(), reason));
	}
	\end{lstlisting}
	This method is used to add funds to a Developer's wallet. It can be called by the Developer himself or by a Supervisor (for a refund). It supports an optional reason for the transaction, which is set to "TOP-UP" if not provided.
	
	\subsubsection{\texttt{charge}}
	\begin{lstlisting}[label={lst:walletcharge}]
	public boolean charge(double amount) {
		if (amount < 0) throw new IllegalArgumentException("Negative amount");
		if (amount == 0) return true;
		if (amount > balance) return false;
	
		this.balance -= amount;
		transactionHistory.add(Transaction.create(
				nextTransactionId++, -amount, LocalDateTime.now(),
				"CHARGE: -" + String.format("%.2f", amount)));
		return true;
	}
	\end{lstlisting}
	This method subtracts \texttt{amount} from the balance. If there aren't enough funds, it simply returns \texttt{false}. Otherwise, it records the transaction and returns \texttt{true}.
	
	
	\subsection{GPU}
	The \texttt{GPU} class simulates a physical GPU running AI models. It implements the \texttt{Subject} (observer design pattern) to notify the \texttt{LoadBalancerService} when the temperature exceeds 90 degrees Celsius. 
	\\The timer sends a tick every 5 seconds to the GPU, which updates its temperature and load percentage through the \texttt{simulateTick} method.
	
	\subsubsection{\texttt{simulateTick}}
	\begin{lstlisting}[label={lst:gpusimulate}]
	public void simulateTick() {
		if (status == GpuStatus.ACTIVE) {
			double increment = 1.0 + Math.random() * 3.0;
			setTemperature(temperature + increment);
			loadPercentage = Math.min(100.0, loadPercentage + Math.random() * 5.0);
		} else if (status == GpuStatus.INACTIVE) {
			if (temperature > 30.0) {
				double decrement = 0.5 + Math.random() * 1.5;
				this.temperature = Math.max(30.0, temperature - decrement);
			}
			loadPercentage = Math.max(0.0, loadPercentage - 5.0);
		}
	}
	\end{lstlisting}
	If the GPU is active, its temperature and load percentage increase. Otherwise, they decrease.

	\subsubsection{\texttt{setTemperature}}
	\begin{lstlisting}[label={lst:gputemperature}]
	public void setTemperature(double temperature) {
		this.temperature = temperature;
		if (temperature > TEMPERATURE_THRESHOLD) {
			this.status = GpuStatus.IDLE;
			notifyObservers("TEMPERATURE_ALERT");
		}
	}
	\end{lstlisting}
	This is the method that notifies the \texttt{LoadBalancerService} when in case of a temperature alert.

	\subsection{GpuCluster}
	This class follows the singleton design pattern and serves as the manager of all the \texttt{GPU}s. It provides a single, globally accessible object for assigning and releasing GPUs across the system.
	\subsubsection{\texttt{getAvailableGpu}}
	\begin{lstlisting}[label={lst:gpuclustergetavailable}]
	public GPU getAvailableGpu() {
	    List<GPU> all = gpuDao.findAll();
	    for (GPU gpu : all) {
	        if (gpu.isAvailable()) {
	            gpu.setStatus(GpuStatus.ACTIVE);
	            return gpu;
	        }
	    }
	    return null;
	}
	\end{lstlisting}
	
	This method returns an available GPU (whose state is set to \texttt{ACTIVE}) that can be assigned to a new session or, in case of load balancing, to an existing one.
	The \texttt{releaseGpu} method simply sets the state of the given GPU to \texttt{INACTIVE}.

	\subsection{Auth Service}
	This class handles the login and the registration of users. 

\subsubsection{\texttt{login} and \texttt{register}}
\begin{lstlisting}[label={lst:authservice}]
public User login(String email, String password) {
	User user = userDao.findByEmail(email);
	if (user != null && user.login(email, password)) {
		return user;
	}
	return null;
}

public User register(String name, String email, String password, String role) {
	if (userDao.findByEmail(email) != null) {
		return null;
	}

	User user;
	switch (role.toLowerCase()) {
		case "developer":
			user = new Developer(nextId++, name, email, password);
			break;
		case "modelprovider":
			user = new ModelProvider(nextId++, name, email, password);
			break;
		case "supervisor":
			user = new Supervisor(nextId++, name, email, password);
			break;
		default:
			return null;
	}

	userDao.save(user);
	return user;
}
\end{lstlisting}
These methods are simple and self-explanatory, but it's worth noticing that the \texttt{login()} method uses the \texttt{login()} method of the \texttt{User} class, which uses hashing with the \texttt{BCrypt} library to verify the password.
\subsection{Complaint Service}
This class handles both the complaint creation by the Developer and the resolution by the Supervisor.

\subsubsection{\texttt{acceptComplaint}}

public void detachGpuFromSession(GPU gpu) {
    for (Session session : sessionDao.findAll()) {
        if (session.isActive() && session.getGpus().contains(gpu)) {
            double lostLoad = gpu.getLoadPercentage();
            session.removeGpu(gpu);
            System.out.println("GPU " + gpu.getId() + " detached from session " + session.getId()
                    + " due to safety/overheating.");
            if (session.getGpus().isEmpty()) {
                System.out.println("Forced termination of session " + session.getId()
                        + " (no GPUs left for developer: " + session.getDeveloper().getName() + ")");
                closeSession(session);
            } else if (lostLoad > 0) {
                double addedLoad = lostLoad / session.getGpus().size();
                for (GPU remainingGpu : session.getGpus()) {
                    remainingGpu.setLoadPercentage(Math.min(100.0, remainingGpu.getLoadPercentage() + addedLoad));
                }
                System.out.println("Lost load (" + String.format("%.1f", lostLoad)
                        + "%) redistributed evenly among surviving GPUs.");
            }
            return;
        }
    }
}
\end{lstlisting}

This method is used to detach a GPU from a session when it is overheated. First it finds the session assigned to the given GPU and removes it from the session. Then it checks if the session still has any GPUs left. If not, it closes the session. If it has GPUs left, it redistributes the load evenly among the remaining GPUs.

\subsection{Load Balancer Service}
This class is responsible for acting as an observer for the \texttt{Timer} and \texttt{GPU} subjects. It uses the strategy design pattern to balance the load among the active GPUs in the cluster with two different possible algorithms.

\subsubsection{\texttt{update} and \texttt{onTemperatureAlert}}
\begin{lstlisting}[label={lst:loadbalancerupdate}]

        GPU gpu = (GPU) subject;

    } else if ("TICK".equals(event)) {

private void onTemperatureAlert(GPU gpu) {
    System.out.println("\n  ALERT: GPU " + gpu.getId()
            + " exceeded 90C! Temperature: " + String.format("%.1f", gpu.getTemperature()) + "C");

    sessionService.detachGpuFromSession(gpu);

    cluster.releaseGpu(gpu);
    System.out.println("GPU " + gpu.getId() + " released and cooling down.");
}
\end{lstlisting}

In the \texttt{update} method, the service checks if the subject is a GPU or a Timer. If it is a GPU, there is an overheating issue and it calls the \texttt{onTemperatureAlert} method. If it is a Timer, it calls the \texttt{balanceLoad} method to rebalance the load.
The timer triggers the load balancing every 5 seconds, whereas a GPU triggers the temperature alert when it reaches 90 degrees.
The \texttt{onTemperatureAlert} method detaches the GPU from the session and releases it. 
The \texttt{releaseGpu} method of the \texttt{GpuCluster} simply sets the GPU status back to \texttt{INACTIVE}, making it available for future sessions once its temperature returns to a safe range.

\subsection{Even Balancing Strategy}
We created this strategy to implement a load balancing algorithm that equally distributes the load between the GPUs.
\subsubsection{\texttt{balanceLoad}}
\begin{lstlisting}[label={lst:evenstrategy}]
public void balanceLoad(GpuCluster cluster, SessionService sessionService) {
    List<Session> activeSessions = sessionService.getAllSessions().stream().filter(Session::isActive).toList();

    for (Session session : activeSessions) {
        List<GPU> gpus = session.getGpus();
        if (gpus.isEmpty()) continue;

        double totalLoad = gpus.stream().mapToDouble(GPU::getLoadPercentage).sum();
        double expectedLoad = totalLoad / gpus.size();

        if (expectedLoad > 80.0) {
            GPU freeGpu = cluster.getAvailableGpu();
            if (freeGpu != null) {
                session.addGpu(freeGpu);
                freeGpu.setStatus(GpuStatus.ACTIVE);
					System.out.println("GPU " + freeGpu.getId() + " activated for load balancing Session "
							+ session.getId() + ".");
                double newLoad = totalLoad / session.getGpus().size();
                session.getGpus().forEach(g -> g.setLoadPercentage(newLoad));
            } else {
					System.out.println(
							"GPU shortage: load will be distributed uniformly ignoring max constraint for Session "
									+ session.getId());
                session.getGpus().forEach(g -> g.setLoadPercentage(expectedLoad));
            }
        } else if (expectedLoad < 20.0 && gpus.size() > 1) {
            GPU removed = gpus.get(gpus.size() - 1);
            session.removeGpu(removed);
            cluster.releaseGpu(removed);
				System.out.println("GPU " + removed.getId() + " detached from Session " + session.getId()
						+ " (Even Balancing).");
            double newLoad = totalLoad / session.getGpus().size();
            session.getGpus().forEach(g -> g.setLoadPercentage(newLoad));
        } else {
            gpus.forEach(g -> g.setLoadPercentage(expectedLoad));
        }
    }
}
\end{lstlisting}

For each session, this method checks the average load. If it goes over 80\%, it tries to add another GPU from the cluster. On the other hand, if the average load is under 20\% and the session has more than one GPU, it removes one. Either way, it always makes sure that the total load is shared equally among the GPUs currently in use.
\subsection{Eco Balancing Strategy}
We created this strategy to implement a power saving load balancing algorithm.
\subsubsection{\texttt{balanceLoad}}
\begin{lstlisting}[label={lst:ecostrategy}]
public void balanceLoad(GpuCluster cluster, SessionService sessionService) {
		List<Session> activeSessions = sessionService.getAllSessions().stream().filter(Session::isActive).toList();

		for (Session session : activeSessions) {
			List<GPU> gpus = session.getGpus();
			if (gpus.isEmpty())
				continue;

			double totalLoad = gpus.stream().mapToDouble(GPU::getLoadPercentage).sum();

			int neededGpus = (int) Math.ceil(totalLoad / 95.0);
			if (neededGpus == 0)
				neededGpus = 1;

            while (session.getGpus().size() < neededGpus) {
				GPU freeGpu = cluster.getAvailableGpu();
				if (freeGpu == null) {
					System.out.println("GPU shortage: cannot handle peak for Session " + session.getId());
					break;
				}
				session.addGpu(freeGpu);
				freeGpu.setStatus(GpuStatus.ACTIVE);
				System.out.println("GPU " + freeGpu.getId() + " activated to handle peak for Session "
						+ session.getId() + " (Eco Strategy).");
			}

			while (session.getGpus().size() > neededGpus) {
				GPU removed = session.getGpus().get(session.getGpus().size() - 1);
				session.removeGpu(removed);
				cluster.releaseGpu(removed);
				System.out.println(
						"GPU " + removed.getId() + " detached from Session " + session.getId() + " to save power.");
			}

			double remainingLoad = totalLoad;
			for (int i = 0; i < session.getGpus().size(); i++) {
				GPU g = session.getGpus().get(i);
				if (i == session.getGpus().size() - 1) {
					g.setLoadPercentage(remainingLoad);
				} else {
					g.setLoadPercentage(95.0);
					remainingLoad -= 95.0;
				}
			}
		}
}

\end{lstlisting}
For each session, this method calculates the total GPU load and figures out how many GPUs are needed, assuming that each one can handle up to 95\% of the load. If the session needs more GPUs, it tries to get free GPUs from the cluster. On the other hand, if it has more GPUs than it needs, it removes the extra ones to save power. Finally, instead of sharing the load equally, it fills the GPUs one by one up to 95\%, and leaves the remaining load on the last one.

	private final Map<Integer, User> storage = new HashMap<>();

	@Override
	public void save(User user) {
		storage.put(user.getId(), user);
	}

	@Override
	public User findById(int id) {
		return storage.get(id);
	}

	@Override
	public User findByEmail(String email) {
		return storage.values().stream()
				.filter(u -> u.getEmail().equals(email))
				.findFirst()
				.orElse(null);
	}

	@Override
	public List<User> findAll() {
		return new ArrayList<>(storage.values());
	}

	@Override
	public void update(User user) {
		storage.put(user.getId(), user);
	}

	@Override
	public void delete(int id) {
		storage.remove(id);
	}
}

If there are no users that match the criteria, the method returns \texttt{null}. This approach avoids the need for less readable and more complex loops.

\section{ComplaintServiceTest}
The Supervisor can also resolve complaints against AI models. In the following test class, we verified the methods \texttt{acceptComplaint()} and \texttt{rejectComplaint()}. First, we instantiated multiple complaints filed by a Developer. What the Supervisor has to check is that, when accepting a complaint, the Developer is refunded correctly and the model's status is updated to blocked.

\begin{lstlisting}[language=Java, caption={Code snippet of the ComplaintServiceTest class}, label={lst:complaint_service_test}]
@Test
void testComplaintAcceptance() {

	assertEquals(2, complaintService.getPendingComplaints().size());

}
